# Player Identity

Empire Streets uses three different identifiers:

- `accountId`: server-verified account identity. It comes from an `AccountIdentityProvider`, not from browser localStorage, player name, email, URL params or `playerId`.
- `playerId`: gameplay identity inside one `serverInstanceId`. It is generated by the server when a registration is created.
- `sessionId`: gameplay API session identity. It is created after a one-time join ticket is consumed.

## Registration

`empire_player_registrations` binds one `accountId` to one `playerId` per `serverInstanceId`.

Required invariants:

- `UNIQUE(server_instance_id, account_id)`
- `UNIQUE(server_instance_id, player_id)`
- client-provided `playerId` is never proof of identity
- concurrent joins for the same account and server resolve to the same registration

The server may accept legacy `playerId` fields for compatibility, but only as deprecated input. Load and command authorization are resolved from the gameplay session.
`LoadGameplaySliceRequest.playerId` is optional and deprecated; it is never used as proof of identity.

## Dev Auth

When no production account provider exists, the dev provider can resolve `accountId` from `x-empire-account-id`, `accountId`, or legacy `playerId`.

This provider is allowed only outside production, or when an explicit dev auth flag is enabled. Production must reject joins without a real account identity provider.
