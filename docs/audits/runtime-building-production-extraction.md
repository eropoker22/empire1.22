## What moved

- Building detail shell creation and modal wiring moved to `ui/buildingDetailPanel.js`.
- Building detail stats, mechanics rows, action rows, empty states and tab syncing moved to `ui/buildingDetailPanel.js`.
- Production building info text/effects/action list rendering moved to `ui/productionPanel.js`.
- Factory info card and factory slot card rendering moved to `ui/productionPanel.js`.
- Pharmacy, drug lab and armory recipe card DOM, quantity controls, input display, progress labels and craft/collect button rendering moved to `ui/recipePanel.js`.
- Building detail pure view-model builders for stats, mechanic rows, action rows, collect/upgrade labels and metadata moved to `runtime/buildingDetailViewModel.js`.
- Building detail info-tab pure rows and action copy moved to `runtime/buildingDetailInfoViewModel.js`; `runtime.js` only maps that view model to DOM nodes.

## What stayed in runtime

Gameplay-coupled code stayed in `runtime.js`:

- production job creation, scheduling, ready-state sync and timer management;
- material/cash validation and consumption;
- inventory/storage mutation and output application;
- factory state sync, slot production toggling, upgrades and collect mutation;
- building detail mechanics resolution, passive production accrual and notification side effects;
- building special action validation, cooldown decisions, state mutation and result feed writes;
- building detail orchestration that reads runtime state, persists passive collect counters and calls `renderBuildingDetailPanel` / info-tab rendering;
- attack, spy, police, ownership and map selection logic.

## Known risks

- Building detail pure view-model creation now lives outside `runtime.js`; runtime still owns the state reads, passive collect persistence, DOM mounting and action callbacks.
- Production cards now rely on callback-driven UI rendering; manual browser checks should focus on button disabled states and quantity controls.
- `client/` remains stale until the publish output is regenerated through the build script.

## Next recommended refactor

Extract another small pure runtime view-model slice with low mutation risk, such as market/player profile read-only view data, while keeping DOM mounting, persistence, action execution, inventory mutation and cooldown updates in `runtime.js`.
